Why another network simulator?

Nab was born out of its authors' inability to simulate large (as in 10000s of nodes) ad hoc networks with existing tools, and impatience in dealing with their internal complexity, which tended to make implementing new functionality a lengthy and bug-ridden task.

Why write it in ocaml?

Ocaml is a high-level, yet highly efficient programming language. It supports both object-oriented and functional programming styles. Ocaml has a number of attributes which make it very well-suited to writing a network simulator:
  • Speed, speed, speed..
  • An interactive toplevel, offering the advantages typically associated with a scripting language (interactively develop scripts, examine simulator state). This makes things much simpler than integrating a scripting language into a non-scripted language (say c++ and otcl), since everything remains in one language domain, with no bindings to write, type conversions, etc.
  • A replay debugger.
  • Support for high-level, structured data types. For example a data type representing an AODV route discovery (sequence of hops + possible route repair floods) can be defined in a few lines.

What are the long-term plans for NAB?

Long-term plans often don't survive the short term, so we'll avoid answering that one (but yes, we do have a few ideas..).

What are the short-term plans for NAB?

In the very near future (ie, early May) we will release an update with a 802.11b-like MAC layer implemented, some propagation models, and a completed implementation of a torus topology.
